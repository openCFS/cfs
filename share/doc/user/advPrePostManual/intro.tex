\subsection{CFS++/NACS HDF5 (.h5) and XDMF2 (.xmf) Files}


\subsubsection{HDF5} The  internal data structures and  temporal succession of
operations  in CFS++ are  most closely  matched by  a hierarchical  data model
\cite{West2011,  DataModelWiki}.    This  can   be  easily  mapped   into  the
hierarchical structure of a HDF5 file.  A number of plugins and extensions for
external codes provide  interfaces to pre- as well  as postprocessors.

 HDF5 is  used as format for  input and output due  to the fact, that  it is a
mature standard. The investigation of  the mesh and therefore visual debugging
before the actual  simulation is possible.  This feature  is especially useful
for  checking the results  of the  intersection calculation  step of  a Mortar
coupled simulation.  The  output of a simulation run can be  used as input for
next simulation,  enabling decoupled simulations.   This can be  necessary for
the  investigation  of  intermediate  result  (esp.   in  aeroacoustics)  HDF5
provides   tested   interoperability   with   other   packages   (Matlab   via
h5read/h5write, Octave  via save -hdf5, Mathematica  via Import/Export, Python
via h5py\footnote{h5py  web site: \url{http://h5py.alfven.org/}},  etc). There
exist also a number of command  line utilities for querying and modifying HDF5
files. For a broader overview cf. Sec. \ref{sec:intro_hdf5tools}.

 Meshes can also  be loaded partially and a global  compuational mesh in CFS++
can be combined from a number HDF5 mesh  files.  A plugin in the form of a VTK
data       reader       is       available       for       ParaView       (cf.
Sec.  \ref{sec:paraview_hdf5_plugin}), which  makes advanced  features  of the
HDF5  datasets accessible.   This includes  better support  for  harmonic data
e.g. animation of harmonic/eigenfrequency  results and the option to visualize
hp-FEM results through the VTK generic adaptor framework \cite{Schroeder2005}.

The  files can  be written and  opened in
the following codes (cf. Fig. \ref{fig:hdf5_xdmf_io}).

\subsubsection{XDMF}
As stated in the abstract of  \cite{Clarke2007}

\emph{The extensible data model and format (XDMF) is an active common data hub
used to  pass meta data and  values between application modules  in a standard
fashion. XDMF  views data  as consisting  of two basic  types: light  data and
heavy data.  Light  data is both meta data and small  amounts of values. Heavy
data  typically consists  of  large arrays  of  values.  Support  for XDMF  is
included in the widely used visualization package ParaView\dots}

The customized HDF5 data model of CFS++ therefore perfectly fits into the more
general XDMF \cite{Clarke2002,  Clarke2007, XDMFWebsite} model, which consists
of slim XML files for describing the structure of the finite element model and
HDF5 files which carry the  large data structures (geometry, mesh, field data,
etc.). This way it is possible to  tune the CFS++ HDF5 format according to the
internal structures within  the program but make the  produced data consumable
by other codes  like ParaView or VisIt (cf.  Sec. \ref{sec:intro_visit}).  The
XDMF format enjoys also growing support in the aforementioned codes, making it
possible  to   use  very  large  files  beyound   four  gigabytes,  on-the-fly
visualization and parallel I/O (cf.  \cite{hpcforgeWebsite}).

Other  advantages  according  to  \cite{ParaViewXdmf}  include,  that  derived
quantities can  be calculated and added  lightly to the data  model.  The Xdmf
model  can reference multiple  HDF5 files.   Each node  of a  running parallel
simulation  can  write  its  own  HDF5  datafile while  an  Xdmf  wrapper  can
concatenate  all pieces.   In transient  simulations the  geometry  can remain
fixed  while  being shared  with  lightweight  links.  The XDMF  library  also
includes converters to the older EXODUS II FEM data model \cite{Schoof1994}.


\subsection{ANSYS Result Files (.rst)}

The .rst  file format is ANSYS  Classic's native output format  for meshes and
simulation   results.    It   is   a  well-documented   binary   format   (cf.
\cite{ANSYSProgrammersManual2009}).  The  files can  be written and  opened in
the following codes (cf. Fig. \ref{fig:rst_io}).

\begin{figure}[htb] 
  \centering 
  \includegraphics[width=0.8\textwidth]{pics/ansys/rst_io}
  \caption{Options for I-DEAS universal data I/O.}
  \label{fig:rst_io}
\end{figure}

\begin{description}

\item[ANSYS Classic] The simplest way  to postprocess results in ANSYS Classic
is by going to Main  Menu $\rightarrow$ General Postproc $\rightarrow$ Results
Viewer. To see the actual APDL  script created by this sequence of commands go
to File $\rightarrow$ List $\rightarrow$ Log File... 

The regions  from a CFS++  simulations get mapped  to material numbers  in the
result files.  CFS++ writes  also a components  (.cm) file which  contains the
real names  of regions and named  entities for postprocessors  that support it
(ANSYS Classic  and CFD-Post).  For further information  please refer  to Sec.
\ref{sec:intro_ansys_classic} and Sec. \ref{sec:postproc_ansys}.

\item[ANSYS CFD-Post] CFD-Post supports loading  of .rst files.  It has strong
support for transient  and static files but seems to lack  the ability to copy
with harmonic  or eigenfrequency  results (c.f. Sec.   8.14.5. ANSYS  Files in
\cite{ANSYSCFDPostUG2010}).   For   further   information  please   refer   to
Sec.\ref{sec:intro_ansys_cfdpost} and Sec. \ref{sec:postproc_cfdpost}.

\item[EnSight]  The dataset can  be broken  down according  to the  regions by
selecting 'Using Material Id' in  the 'Regular Part Creation Convention' field
of the  Format options tab  of the open  dialog. Advanced options  for loading
.rst  datasets into  EnSight  can be  found  at the  software  website of  the
National Energy Research  Scientific Computing Center (NERSC)\footnote{Loading
ANSYS             .rst             files             into             EnSight:
\url{http://www-vis.lbl.gov/NERSC/Software/ensight/doc/OnlineHelp/UM-C2.pdf}
p.                                                                        2-19,
\url{http://www-vis.lbl.gov/NERSC/Software/ensight/doc/OnlineHelp/HT-Read-ANSYS.pdf}}
For  further  information  refer  to Sec.   \ref{sec:intro_ensight}  and  Sec.
\ref{sec:postproc_ensight}.

\item[Tecplot] ANSYS .rst files can be  opened in Tecplot 360 by going to File
$\rightarrow$   Load  Data  File(s)...   $\rightarrow$  ANSYS   Results  (FEA)
$\rightarrow$           OK.            Sec.           \ref{sec:intro_tecplot},
Sec. \ref{sec:postproc_tecplot}

\item[unv2rst] The unv2rst  tool can be used to  convert IDEAS universal files
(cf.  Sec.  \ref{sec:intro_unv}) written by  CAPA, CFS++ or NACS to .rst files
compatible    with    versions    5.7\footnote{ANSYS    5.5    .rst    format:
\url{http://www.ansys.stuba.sk/html/prog_55/g-int/INS1.htm}},  6.0  (default),
6.1 and  later ANSYS versions.   Please note, that  the unv2rst tool  seems to
produce only  the correct release date  for 6.0. Please mind,  that the native
API for writing of  these older .rst files before release 8.1  has used 32 bit
integers for  internal pointers and  therefore some restrictions in  file size
may apply. Since ANSYS release 8.1 all  .rst API calls use 64 bit integers for
internal pointers (cf.  Sec.  1.2.3 \cite{ANSYSProgrammersManual2009}) and the
API functions have an 8 as  suffix since that release (e.g. binhed, binwrt are
now binhed8 and binwrt8).

\item[rstinfo]  We  provide the   script {\tt  share/scripts/rstinfo}
which  prints out informations  like the  ANSYS release  the file  was created
with. This  way it  is possible to  determine if  the file can  potentially be
opened with a given postprocessor.

\end{description}

ANSYS .rst  files can  either be accessed  directly or  by using a  binary API
provided by ANSYS.  The unv2rst tool from CAPA\footnote{CAPA is the commercial
predecessor  of   CFS++/NACS.  An   information  brochure  is   available  at:
\url{http://www.capanet.de/download/pdf/capa.pdf}},  implements access  in the
former way, whereas the .rst writers in CFS++/NACS build upon the ANSYS API.

The writers in CFS++ support writing .rst files compatible with ANSYS versions
10.0 until 13.0 (can be adjusted  using the ansysRevision attribute in the rst
XML input tag).

EnSight  and  Tecplot.  The  same  is naturally  true  for  older releases  of

(cf.      Tab.       \ref{tab:ansys_releases}),     EnSight     (cf.      Tab.

release  histories in order  to find  out if  a given  program version  may be
potentially compatible with a given .rst file.

\subsection{I-DEAS Universal Files (.unv) and CAPA (.unverg) Files}
\label{sec:intro_unv}

The  I-DEAS universal  file  format is  a  general format  originally used  by
SRDC's\footnote{Structural        Dynamics        Research        Corporation:
\url{http://en.wikipedia.org/wiki/SDRC}} I-DEAS\footnote{Integrated Design and
Engineering   Analysis   Software:  \url{http://de.wikipedia.org/wiki/I-DEAS}}
Master Series  software to  encode CAD/CAM models  and FEM  simulation results
(cf.  Fig.   \ref{fig:unv_ideas}).  The I-DEAS software is  still published by
Siemens    PLM   Software\footnote{Siemens    NX    PLM   software    website:
\url{http://www.siemens.com/plm}} as part of their NX CAD/CAM/CAE solution.

and FEM  cell types. It  is an  open ASCII datafile  format, which is  used by
several   finite    element   codes.    It   is    divided   into   individual
datasets\footnote{A complete description of  all universal dataset numbers can

\url{http://www.sdrl.uc.edu/universal-file-formats-for-modal-analysis-testing-1/file-format-storehouse/file-formats}

contain  certain informations. The  application can  decide which  datasets to
take into account and which ones to disregard.

   The   files   can  be   opened   or   created   in  the   following   codes
(cf. Fig. \ref{fig:unv_io}):

\begin{description}

\item[Gmsh] Gmsh  can read and write .unv  meshes but no data  fields. It does
not   respect   the   group   definitions   given  in   CFS++   output.   Sec.
\ref{sec:intro_gmsh},               Sec.               \ref{sec:preproc_gmsh},
Sec. \ref{sec:postproc_gmsh}

\item[GiD]       GiD       can       write       .unv       meshes       using
Files$\rightarrow$Export$\rightarrow$Using Template...$\rightarrow$UNV.

\item[unv2rst]  The software CAPA  also generates  universal file  output, but
uses  different  datasets  for  elements  and  nodes.  Also  some  conventions
regarding  the intepretation of  plane coordinates  are different.  CFS++ also
supports  output  in the  CAPA  .unverg  format.   The files  generated  using
CAPA/CFS++ can indeed be read by  Gmsh, EnSight and Tecplot but no geometry or
result information can be obtained.

\item[Matlab]    UFF   bundle\footnote{UFF    File   Reading    and   Writing:
\url{http://www.mathworks.com/matlabcentral/fileexchange/6395-uff-file-reading-and-writing}}
contains 2 functions, one for reading (readuff) and one for writing (writeuff)
of ascii and  binary files in Universal File Format  (UFF). Although there are
many different  data-sets that  can be  stored in UFF  files in  general, only
data-sets 58,  58b, 15, 82, 2411, 2412,  151, 164, and 55  are supported (2411
and 2412 for reading only).  These data-sets are frequently used in structural
dynamics.

\item[EnSight] The EnSight postprocessor  does not automatically recognize the
required reader for  .unv files. Therefore, the 'SDRC Ideas'  format has to be
selected in  the Format field  of File $\rightarrow$ Open...   dialog. EnSight
respects  the element  group  descriptions (regions  and  named elements)  but
disregards the node groups (named nodes).  The dataset can also be broken down
according to the regions by selecting 'Using Material Id' in the 'Regular Part
Creation Convention' field of the Format options tab of the open dialog.  Sec.
\ref{sec:intro_ensight}, Sec. \ref{sec:postproc_ensight}

\item[Tecplot] I-DEAS .unv files can be opened in Tecplot 360 by going to File
$\rightarrow$ Load  Data File(s)...  $\rightarrow$  SRDC/IDEAS Universal (FEA)
$\rightarrow$           OK.            Sec.           \ref{sec:intro_tecplot},
Sec. \ref{sec:postproc_tecplot}

\item[CFX-Pre]  I-deas  universal files  can  be used  as  inputs  in the  CFX
preprocessor (\cite{CFXPreUG2009} p. 60).

\item[ANSYS FLUENT] You can import  an I-deas Universal file into ANSYS FLUENT
in three  different ways\footnote{Import an  I-deas Universal file  into ANSYS
FLUENT: \url{https://www.sharcnet.ca/Software/Fluent12/html/ug/node174.htm}}.

\begin{itemize}
\item  Generate  an  I-deas  surface  or volume  mesh  containing  triangular,
quadrilateral, tetrahedral,  wedge and/or hexahedral elements.  Import it into
TGrid using  the commands described in  the TGrid User's Guide.  Adhere to the
restrictions  described in Appendix  B of  the TGrid  User's Guide.  In TGrid,
complete the  mesh generation  (if necessary) and  follow the  instructions in
Section 6.3.3.

\item Generate  an I-deas volume  mesh with linear  triangular, quadrilateral,
tetrahedral,  wedge, or  hexahedral  elements. Import  it  directly using  the
File/Import/I-deas Universal... menu item, as described in Section 4.12.10.

\item Generate  an I-deas volume  mesh with linear  triangular, quadrilateral,
tetrahedral, wedge, or  hexahedral elements. Use the fe2ram  filter to convert
the Universal  file to the  format used by  ANSYS FLUENT. To convert  an input
file in  I-deas Universal  format to  an output file  in ANSYS  FLUENT format,
follow  the instructions below  in Section  6.3.11. After  the output  file is
written, read it  into ANSYS FLUENT using the  File/Read/Mesh... menu item, as
described in Section 4.2.
\end{itemize}

\item[Siemens NX PLM] Since NX is the successor of the original IDEAS software
it should  be possible to read .unv  files.  Since the workflow  within the NX
system is tightly  integrated, it is not easily possible  to read an arbitrary
.unv file into this environment for postprocessing. The user has to define the
model geometry, assign a corresponding  FEM model consisting of mesh, boundary
conditions  and materials  and after  that a  simulation (e.g.  ANSYS, ABAQUS,
Ideas,  etc.)  has  to be  defined.  The  user can  then choose  to  start the
simulation from within NX or to write  input files for the solver. In the case
of an  Ideas simulation, it  just writes  an .unv file  with mesh and  BCs.  A
third party code like CFS++ would then  read this .unv file and write back the
results into an  .unv file with the same nodes and  elements. This result file
could then  be imported into  the NX model  for postprocessing.  This  has not
been tested though.


\item[Amira] In  Amira\footnote{Amira website: \url{http://www.amira.com}} the
following  subset of  so-called universal  dataset numbers  is  supported when
importing I-DEAS files:

\begin{itemize}
\item 15: Nodes with single precision coordinates.
\item 781, 2411: Nodes with double precision coordinates.
\item  71, 780, 2412:  Cell definition.  Lines, triangles,  quads, tetrahedra,
hexahedra, and  prisms with  linear shape functions  are supported.  Quads are
decomposed   into  triangles.   Prisms  are   decomposed  into   triangles  as
well. Hexahedra are decomposed into  tetrahedra if tetrahedra or prisms appear
too (Amira currently is not able  to handle meshes with mixed element types or
elements with higher-order shape functions).
\item 55, 2414: Data at nodes.
\end{itemize}  The common  file name  extension of  this format  is  .unv. For
I-DEAS files  with some other  extension the file  format has to  be specified
manually via the  popup menu of the file  dialog. Surfaces, tetrahedral grids,
and hexahedral grids can also be exported into an I-DEAS universal file.

\item[SALOME]  The  CAD/mesh   open-source  software  SALOME  \footnote{SALOME
website:  \url{http://www.salome-platform.org}}  is  intended to  create  .unv
files.

\item[OOFEM] There exists a  Python based converter\footnote{OOFEM .unv Python
converter:     \url{http://www.oofem.org/wiki/doku.php?id=unv2oofem:unv2oofem}}
inside  the OOFEM  project for  reading and  converting .unv  files .   It can
easily be extended for other output formats.

\item[Code  ASTER] Code  ASTER has  also  an interface  for reading  universal
files\footnote{Code            ASTER            .unv            Preprocessing:
\url{http://caelinux.com/downloads/docs/PRE_IDEAS_EN.pdf}}
\end{description}

\subsection{CFD General Notation System (.cgns) Files}

For  the import  of CFD data  CGNS is  the most general  choice of
formats.  It defines a data  model (Standard Interface Data Structures (SIDS))
as well as data  format with underlying ADF\cite{Dickens1998,CGNSADFManual} or
HDF5 layers.

According to \cite{CGNSSIDSManual}, a  CGNS database can contain the following
information:  grid coordinates  and  elements, flow  solution data,  including
nondimensional   parameters,  multizone   interface   connectivity,  including
abutting  and  overset,   boundary  conditions,  flow  equation  descriptions,
time-dependent    flow,    reference    states,    dimensional    units    and
nondimensionalization information  associated with data,  convergence history,
association  to  geometry  definition  and  topologically  based  hierarchical
structures

The format  can be written  by a  wide number of  CFD codes and  therefore the
aeroacoustic environment developed in this thesis  can be tested with a lot of
different input data sets. Although intended for CFD computations, application
codes can also save different physical  field data to CGNS. Qualifying it as a
future  candidate  for a  second  data  hub,  especially for  commercial  post
processors  as Fig.  \ref{fig:cgns_io}  shows.   The files  can  be opened  or
created in the following codes:

\begin{figure}[htb] 
  \centering 
  \includegraphics[width=\textwidth]{pics/cgns/cgns_io}
  \caption{Options for CGNS data I/O.}
  \label{fig:cgns_io}
\end{figure}

\begin{description}

\item[cplreader]  The CAA coupling  reader of  CFS++ can  read CGNS  files and
generate acoustic source terms from CFD fields (cf. \cite{cplreadermanual}).

\item[EnSight] The CGNS reader inside EnSight does not seem to be very robust.
Find  more information  in  the readme  file\footnote{CGNS reader  README.txt:
\texttt{  /path/to/ensight/CEI/ensightXX/src/readers/cgns/README.txt}}  and in
the documentation of the corresponding source code.

\item[Tecplot]  CGNS files  can be  opened  in Tecplot  360 by  going to  File
$\rightarrow$  Load Data File(s)...   $\rightarrow$ CGNS  Loader $\rightarrow$
OK. The zones and  variables to be load can be specified  in the Options field
of      the      loader      dialog.       Sec.       \ref{sec:intro_tecplot},
Sec. \ref{sec:postproc_tecplot}

\item[CFX-Pre] CGNS  files can be used  as inputs in the  CFX preprocessor and
CFX-Pre can export to CGNS files (cf.  \cite{CFXPreUG2009} p.  53 and p. 164).
After  a simulation  run the  results  can be  converted into  CGNS using  the
cfx5export (cf. \cite{CFXSolvManUG2009} p. 143) tool.  ANSYS CFD-Post can also
read CGNS  files (cf. \cite{ANSYSCFDPostUG2010} p. 114).  A short introduction
German    is   available   at    KIT\footnote{CFX   Kurzanleitung    at   KIT:
\url{http://www.scc.kit.edu/produkte/3852.php}}

\item[FLUENT]  The current  release of  FLUENT allows  you to  export  data to
ABAQUS,    ANSYS,    ANSYS    Input,   CGNS\footnote{FLUENT    CGNS    export:
\url{https://www.sharcnet.ca/Software/Fluent12/html/ug/node128.htm}}, EnSight,
FIELDVIEW, I-deas, Tecplot and other formats.

\item[FieldView]  FieldView  is  powerful  CFD post-processing  software  that
quickly  identifies  important  flow  features  and  characteristics  in  your
simulations. It  allows interactive exploration for  thorough understanding of
your  results.  It also  supports  CGNS\footnote{FieldView supported  solvers:
\url{http://www.ilight.com/solvers.php}}

\item[Pointwise]             Pointwise\footnote{Pointwise             website:
\url{http://www.pointwise.com/}}  is a  software solution  to the  top problem
facing engineering analysts: mesh generation for computational fluids dynamics
(CFD).  CGNS is Pointwise's native data format.

\item[OpenFOAM] OpenFOAM supports CGNS through two dedicated import and export
tools\footnote{cgnsToFoam                    and                   foamToCGNS:
\url{http://openfoamwiki.net/index.php/Sig_Turbomachinery_Utilities_CGNS_Converters}}

\subsection{Gmsh}
\label{sec:intro_gmsh}
 Gmsh is  a 3D finite element grid  generator with a built-in
CAD engine and post-processor. Its design goal is to provide a fast, light and
user-friendly meshing  tool with  parametric input and  advanced visualization
capabilities. Gmsh  is built around  four modules: geometry, mesh,  solver and
post-processing.  The specification  of any  input  to these  modules is  done
either interactively using the graphical user interface or in ASCII text files
using Gmsh's own scripting language.

The  main advantage of  Gmsh is  that is,  small, fast  (at least  for smaller
models) and free  of charge.  If you have  bootstrapped your machine according
to  Sec.    2.1  in  \cite{CFSDevelManual},  you  should   already  have  Gmsh
installed. Otherwise  go to  the official download  site\footnote{The official
Gmsh download  site is: \url{http://geuz.org/gmsh}}. Since CFS++  can read the
Gmsh mesh format and write into the Gmsh postprocessing format if USE\_GMSH is
switched ON in the CMake configuration, no additional extensions are required.

Please  refer to sections  \ref{sec:gmsh_pre} and  \ref{sec:postproc_gmsh} for
informations about pre- and postprocessing  of CFS++ meshes and results and to
Sec. \ref{sec:cfs_tool}  for the possible  ways of data conversion.  The CFS++
entities  (regions, named  nodes and  named elements)  are mapped  to physical
entities in Gmsh.

\subsection{GMV  -  General  Mesh  Viewer}

GMV  is an  easy to  use, 3D-scientific  visualization tool  designed  to view
simulation data from any kind of  structured or unstructured mesh. GMV is free
software  but not open  source (Terms  of use)  and was  developed at  the Los
Alamos National  Laboratory.  The GMV postprocessor is  not publicly available
any  more, but  is still  installed  at Erlangen  and Klagenfurt.   It may  be
obtained also from the CFS++ development server.

\noindent It has the following features:
\begin{itemize}
    \item Read and display cell geometry
    \item Display nodes, cells, surfaces and particle tracers
    \item Isosurface and cutplane interpolation of grid data
    \item Ability to isolate a cell or groups of cells for closer analysis
    \item A wide variety of animation options
    \item A flight mode for interactive fly-throughs of the simulation
    \item Many convenience features for high qualit video production
    \item (A highly intuitive and well behaved user interface)
    \item Visualizes data from any current or future 3-D simulation 
\end{itemize}

\subsection{Octave}

\begin{verbatim}
d = load('-hdf5', 'check.h5')
save -hdf5 out.mat
\end{verbatim}

Octave legt die Variablen in Groups ab und darin erst die Datensaetze:

\begin{verbatim}
octave:1> x = diag ([1,2,3]);
octave:2> x
x =

   1   0   0
   0   2   0
   0   0   3

octave:3> save -hdf5 foo.mat x
octave:4> system("h5ls foo.mat/x")
type                     Dataset {SCALAR}
value                    Dataset {3, 3}
ans = 0
octave:5> system("h5dump -d /x/value foo.mat")HDF5 "foo.mat" {DATASET "/x/value" {
   DATATYPE  H5T_IEEE_F64LE
   DATASPACE  SIMPLE { ( 3, 3 ) / ( 3, 3 ) }
   DATA {
   (0,0): 1, 0, 0,
   (1,0): 0, 2, 0,
   (2,0): 0, 0, 3
   }
}
}

ans = 0
\end{verbatim}

Dritte Spalte einlesen:

\begin{verbatim}
octave:14> system("h5dump -d /x/value -s 0,2 -S 1,1 -c 3,1 -k 1,1 foo.mat")
HDF5 "foo.mat" {
DATASET "/x/value" {
   DATATYPE  H5T_IEEE_F64LE
   DATASPACE  SIMPLE { ( 3, 3 ) / ( 3, 3 ) }
   SUBSET {
      START ( 0, 2 );
      STRIDE ( 1, 1 );
      COUNT ( 3, 1 );
      BLOCK ( 1, 1 );
      DATA {
      (0,2): 0,
      (1,2):  0,
      (2,2):  3
      }
   }
}
}
ans = 0
\end{verbatim}
