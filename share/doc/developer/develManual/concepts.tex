\subsection{Design Goals}

\begin{description}

\item[Manual  Convertibility] It  should  be easily  possible  to copy  around
result datasets between files which contain the same mesh, e.g. in order to do
comparative  studies  (e.g.   in  ParaView).   E.g.   just  the  corresponding
ResultDescription  group and  the  groups under  the  Step groups  have to  be
copied. The h5tool makes it  possible to change groups, datasets or attributes
from the command line.

\item[Everything in  one Place]  After having completed  a simulation  as much
information  about the  simulation run  (e.g. input  XML file,  material file,
machine information,  information about  other simulations after  a conversion
step from CFX  or OpenFOAM, etc.) as possible should be  available in a single
place.  This is  achieved by  attaching the  corresponding information  to the
/UserData group.

\item[Restart from Result  File] Especially on batch processing  systems it is
necessesary  for a numerical  code to  have restart  capability. This  is true
because  computational resources  are assigned  to processes  in a  time slice
scheduling process. In order to  implement restart capability it is necessary,
to not  only store the  primary result variable,  but also its  different time
derivatives according  to the time  discretization scheme applied.   By saving
this information also into the HDF5  file it becomes also possible to evaluate
these quantities in  a post-processor. The HDF5 format  makes it also possible
to  move restart  files between  different system  architectures  (e.g. little
vs. big endian machines).

\item[Native  Data Hub]  The HDF5  files are  considered CFS++'s  native files
because they support every available  feature present in CFS++. These features
contain the  ability to read/write the  mesh, the mesh based  result data, the
history data which is restricted to  some entities in the mesh. All other pre-
and post-processing data  formats only support a subset  of these features. By
supporting  all  features   it  is  also  possible  to   use  HDF5  files  for
comparisons. E.g. the whole test suite builds upon comparisons between results
or history results in reference HDF5 files and HDF5 files generated by running
simulations. For a format conversion matrix refer to Sec. \ref{sec:cfs_tool}.

\item[Constricted File Systems] Simulation  data has to be transferred between
different      machines.      Nowadays,      USB     devices,      CDs     and
DVDs\footnote{\url{http://en.wikipedia.org/wiki/ISO\_9660\#The_2.2F4_GiB_file_size_limit}}
or similar  media are the preferred  means for data  transfers.  These devices
are however often formatted using FAT32  which restricts the file size to 2GB.
In order to  support such file systems the  transient/harmonic datasets can be
saved to external files which prevents the main file from growing beyond 2GB.

\item[Simple Hierarchy] Some  tools for reading HDF5 files  do not support all
features.  Therefore the  data is saved in a very simple  hierarchy, therewith
avoiding the use of complex database-like tables containing composite data.

\item[Long  Term Archivability]  The  everything in  one  place paradigm  also
ensures that data  in HDF5 files can  be interpreted a long time  after it has
been generated. The mature nature of  the HDF5 format and long term support of
the project  developers also  contribute to this  goal. A  lot of time  can be
saved for scientists who have to continue on research projects from colleagues
if  they can  easily reuse  old data.  The goal  of archivability  of research
results  is  therefore even  formulated  in  guide  lines for  making  project
proposals for research foundations like the  DFG (cf. Sec.  3.7 Umgang mit den
im Projekt erzielten Forschungsdaten in \cite{DFGGuidelines}).

\end{description}
