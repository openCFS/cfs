In order to keep a large  software project with many contributors at different
coding  skill levels in  a healthy  state and  to prevent  it from  becoming a
so-called                  monster                  code                  (cf.
\url{http://www.nature.com/news/2010/101013/full/467775a.html})  an  elaborate
infrastructure has to be set up.  This infrastructure includes version control
systems,  build  scripts  for   external  libraries,  good  documentation  and
automated  testing  facilities.  Especially  the  last  requirement cannot  be
overrated.  It  is a means  of maintaining a  basic principle in  science: the
reproducibility of results.

It is the intention of this manual to describe not only the mentioned parts of
the infrastructure  but also to give  an overview of the  internal workings of
CFS++.

\subsection{gridlib}  In  the beginning  of  CFS++  development a  cooperation
between  the chair  of computer  graphics (LGDV)  in Erlangen  in the  area of
adaptive grid  refinement, resulted  in the coupling  of CFS++ to  the gridlib
\cite{Kipfer2003, gridlib, gridlib2001} library.  The goal of this library was
to  provide  grid management  features  tailored  to  the needs  of  numerical
simulation,  as   well  as  for  scientific   visualization.   A  socket-based
communication system between  CFS++ and gridlib was implemented  to review the
incremental performance  of adaptive simulations  in \cite{Triebenbacher2003}.
Also new algorithms for slice  calculations were implemented into gridlib (cf.
Fig.   \ref{fig:gridlib}).    After  the  main  developer   of  gridlib  Peter
Kipfer\footnote{\url{http://www.kipfer.de}}   left  LGDV   for   his  post-doc
position   at   TUM\footnote{\url{http://wwwcg.in.tum.de/people/Alumni}},  the
gridlib project slowly died however.

\subsection{CFSPP} During a HiWi project at  LSE in the years 2004 and 2005 an
attempt was made to implement  a first VTK-based postprocessor for CFS++.  Its
name                                                                        was
CFSPP\footnote{\url{https://lse17.e-technik.uni-erlangen.de:2001/files/precompiled/legacy/cfspp}},
which   stood   for   CFS++   Post-Processor.    It   was   built   upon   the
ORCAN\footnote{\url{http://orcan.sourceforge.net/}}   framework   for  modular
application  development.    This  was  a  cooperation   project  between  the
Fraunhofer  Institute  IIS-B, and  a  few  departments  of the  University  in
Erlangen.   CFSPP implemented  a  few  features, which  are  now available  in
ParaView. Most  notably it supported switching  regions on and  off, which was
not  directly supported by  VTK at  that time.   CFSPP was  even used  für the
excercise classes of two courses in Erlangen.  It was however abandoned, since
only one  person was developing it,  its terse user documentation,  and due to
the fact  that some people  just could  not agree with  the sound of  its name
(pronounced: [si:æf:æs:pi:pi]).   But valuable  experience with VTK  and CMake
was gathered  during this time.   Figure \ref{fig:cfspp} shows the  CFSPP GUIs
running on  Windows and  Linux.  The panels  for the  visualization parameters
should look strangely familiar to a ParaView user.

\subsection{The  OLAS  Machete  Branch}  At  the  same  time  when  CMake  was
introduced  to CFS++,  it was  also  decided that  the previously  independent
linear algebra system should be merged  into the CFS++ source tree. There were
some hurdles for doing so however
\begin{itemize}
\item  OLAS was  following a  mathematicians' or  Fortran-based  philosophy of
using  one-based  array  indices  instead  of  zero-based  ones  like  C-style
languages.
\item OLAS was built as extra library  and therefore had it's own way of doing
I/O and error handling.
\item OLAS  used its  own matrix  and vector classes,  which were  not readily
compatible with the ones in CFS++. Therefore the data exchange between the two
systems was based on pure C-style arrays.
\end{itemize}

\noindent Due to these  points a serious amount of work had  to be invested to
thin  out the  redundant structures  within the  code.  After  a visit  to the
cinema in fall 2007, where Gerhard Link, Simon Triebenbacher and the initiator
Ludwig Bahr  took part, the  fitting name for  the working branch  was finally
found. At the beginning of  the well-known zombie B-movie 'Planet Terror' (cf.
Fig. \ref{fig:planet_terror}  and \url{http://www.imdb.com/title/tt1077258}) a
fake   trailer  for  a   future  movie   (cf.   Fig.    \ref{fig:machete}  and
\url{http://www.imdb.com/title/tt0985694}) was  shown, where a  former Mexican
federal agent  imposes law and order  on corrupt Texan  officials and ruthless
Mexican drug lords using his -- MACHETE!

\subsection{NACSDEPS} The idea  for NACSDEPS was born after  the first version
of NACS  was finally running  on Windows (cf. Fig.  \ref{fig:nacs_win}).  Then
the  need   to  setup  more  Windows  development   machines  within  Simetris
GmbH\footnote{\url{http://www.simetris.de}}  arose and  therewith the  need to
know  how  all the  libraries  on  which NACS  depends  have  to  be built  or
installed.   After the  painful experiences  gathered, while  getting  all the
libraries for the Windows port to work  on a single machine, it was decided on
a weekly  NACS meeting, that  the build process  of these libraries  should be
thoroughly documented.   In the next weeks  the idea of  documenting the build
process by  providing automated build scripts  grew.  By doing  so, we finally
ended up with a system, which  does not only build all our dependent libraries
on Windows but also on all  other supported platforms.  In the first revisions
of NACSDEPS  we also provided prebuilt  binaries for a  few platforms directly
through  the  Subversion server.   It  became  however  quickly unfeasible  to
provide binaries for all our platforms and the time and bandwidth requirements
needed for a checkout just became to high.  Due to this, we decided, that only
the  build descriptions,  checksums and  patches  should be  contained in  the
central repository whenever  possible and that sources should  be fetched from
the internet and binaries should be cached locally instead of centrally.
