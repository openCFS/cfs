\section{History of CFS++}

The Finite-Element (FE) software has  been named {\sf CFS++} according to {\sf
C}oupled  {\sf F}ield  {\sf S}imulation  and the  {\sf ++}  indicates  that as
programming languange C++  has been chosen. Originally, the  first struture of
CFS++ has been designed by  M.~Kaltenbacher and S.~Reitzinger during a stay of
M.~Kaltenbacher in Linz, Austria in the year 2000 at the SFB013 (Numerical and
Symbolic  Scientific Computing).  The  idea was  to  set up  a new  simulation
software for scientific computing capabale for multi-field applications. Since
then on the  group of M.~Kaltenbacher at the  Department of Sensor Technology,
University of  Erlangen-Nuremberg in  Germany was growing  and the  ideas were
realized  in  the  FE  software  CFS++.  In  2008,  M.~Kaltenbacher  and  many
co-workers moved  to University Klagenfurt, Austria.   Furthermore, within the
Cluster  of  Excellence  for  Engineering  of  Advanced  Materials  (niversity
Erlangen-Nuremberg, Germany),  the research  group of M.~Stingl  contribute to
CFS++ concerning shape and topology optimization.

Among many scientists,  who have contributed to CFS++,  we want to acknowledge

\section{Structure of CFS++ and Programming Environment}
\label{Sec:CFSstructure}

From the  beginning on, the main focus  was and still is  on numerical schemes
for  solving   the  arising  system  of  PDEs   within  multi-field  problems.
Therewith,  we  have  no  capabilities  concerning  geometric  modeling,  mesh
generation or post-processing. Instead, we  provide a variety of interfaces to
obtain   the   appropriate   input   data   from   a   number   of   supported
pre-processors\footnotemark[1], as well as  to write different file formats to
feed a variety of post-processing tools\footnotemark[2].

\subsection{Data I/O and Class Structure }
\noindent
To obtain the necessary input information for our numerical simulation
tool a number of input files need to be provided: \\


\noindent {\it  Mesh and field data:}  This data provides  the coordinates and
mesh topology  of all 3D  (volume), 2D (surface/interface) and  1D (interface)
elements. Due  to the fact, that all  elements are tagged with  unique ids, we
can  differ  between  different  physical  regions  where  different  material
properties may  be applied, interfaces  for the specification of  couplings as
well  as surfaces within  the computation.   Furthermore, it  provides special
elements and nodes  on which boundary or/and load  conditions can be specified
or which are used as monitoring points for saving special physical data (e.g.,
acoustic  pressure over  time).  In  addition to  the mesh  information, field
information  may be  read  as input  for  a simulation  (e.g. acoustic  source
terms).  By default, this data is  stored in the hierarchical file format HDF5
\cite{hdf5}. However, we also support a number of different pre-processors and
data  sources  \footnote[1]{Supported  pre-processors  and  CFD  codes:  ANSYS
Classic, GiD, Gmsh, OpenFOAM, FASTEST, ANSYS-CFX}. \\

\noindent {\it Physical problem description:}  To have a standardized and easy
extendable file  format, we use for  this task the  XML-format.  By specifying
schemas \cite{xmlschema}  for our  expected input formats,  it is  possible to
validate  input files  written by  humans as  well as  automatically generated
ones.

\noindent Concerning the result data,  our standard file format is again HDF5.
This enables  us to do chained  simulations, since the output  of a simulation
may  be used  as  an  input for  a  subsequent simulation.   We  may also  use
third-party tools  like Matlab in  this chain, due  to the fact that  the HDF5
format  is  supported  by  many  tools.  Furthermore,  as  displayed  in  Fig.
\ref{fig:cfs_IO}, we provide many tool specific input- and output-file formats

The whole  software is written in {\sf  C++} and we exploit  its class concept
without  loosing   numerical  performance   by  using  techniques   like  e.g.
expression  templates.   The  goal  was  and  still  is,  that  our  numerical
simulation tool  {\sf CFS++} can be easily  extended for any new  PDE, any new
physical coupling as well as  new types of finite elements (e.g, discontinuous
finite elements).

\subsection{Development Environment}
\noindent
%One of the most important points, when many persons collaborate on the
%same program, is to have an advanced development environment.

Concurrent  development of  a large  software project  by  several programmers
requires   an   advanced  development   environment   and  agile   programming
practices\footnote{Agile                 Software                 Development:
\url{http://www.agile-process.org/}} need  to be  applied.  We use  a standard
infrastructure of tools helping us to maintain a version controlled\footnote{A
Visual             Guide            to             Version            Control:
\url{http://betterexplained.com/articles/a-visual-guide-to-version-control}\\Revision
Control:      \url{http://en.wikipedia.org/wiki/Revision_control}}      source
repository,  defined   builds  of   external  packages,  an   automated  build
system\footnote{Build System: \url{http://en.wikipedia.org/wiki/Build_system}}
and        continuous       integration\footnote{Continuous       Integration:
\url{http://www.martinfowler.com/articles/continuousIntegration.html}} is used
on a range of computer platforms used by developers and users.

 To catch integration  errors, we have established nightly  builds. For these,
the  latest source  code  is  checked-out from  our  central code  repository,
compiled,    and    over    hundred   regression\footnote{Regression    Tests:
\url{http://en.wikipedia.org/wiki/Regression_testing}}         and        unit
tests\footnote{Unit                                                      Tests:
\url{http://www.extremeprogramming.org/rules/unittests.html}}  are  run.   The
results of  these tests  are posted  to a central  dash board  and in  case of
errors  or warnings,  emails are  sent to  the developer's  who  committed the
latest changes to the code. As  a side-effect of the nightly builds, we always
have the  latest executables at hand  and distribute them to  a common network
share, so that they can be accessed from any workstation at our institutes.

  Besides  personal  communication  and   developers'  meetings,  we  apply  a
web-based    project   management    and    bug-tracking   tool\footnote{Trac:
\url{http://en.wikipedia.org/wiki/Trac}}  which also  features  interfaces for
discussions               and               a              Wiki\footnote{Wiki:
\url{http://en.wikipedia.org/wiki/Wiki}}. This helps  us to coordinate further
developments.   Despite  all these  efforts,  our truck  number\footnote{Truck
Number:   \url{http://c2.com/cgi/wiki?TruckNumber}}  is   unfortunately  quite
low. This is not uncommon for  software projects in academia and the intention
of this manual is therefore, to change this situation for the better.

We utilize the following tools:\\

\noindent
{\sf SVN:} Subversion is a free and open source version control
  system, which manages files, directories, and the changes made to
  them over time. It facilitates the recovery of older versions and the
concurrent management of  different developing branches \cite{svn}.\\

\noindent {\sf CFSDEPS:} When developing a general purpose finite element code
a lot  of external libraries  (e.g.  BLAS, LAPACK,  ARPACK, Metis) have  to be
included.   Many  of these  libraries  are  present  on modern-day  Linux/Unix
distributions.  If a library is however not available, it has to be built from
source. Since more  than the standard GNU compiler  suite should be supported,
it is  almost certain that  incompatibilities with pre-built  system libraries
will  arise.   Therefore, we  decided  to build  all  our  dependencies in  an
automated script  based manner.  The scripts  fetch the sources  from the web,
unpack them, perform a checksum check, patch, configure, build and install the
libraries.  This  way we obtain  maximum platform-independence while  having a
well-documented  and  SVN-versioned  description  of  how  to  build  external
packages. In  the past the CFSDEPS  were located in a  separate SVN repository
and  Perl scripts were  used to  do platform  independent builds  at configure
time. In the latest revisions, the CFSDEPS have been moved into the main CFS++
repo and CMake external  project builds (cf. \cite{BuildExtPackages}) are used
at build time.  \\

\noindent
{\sf CMake Build System:}  CMake is a family of platform
independent tools designed to  build, test and package software
\cite{Martin2007a}. %cmake, Schroeder2004, Martin2007a}. 
It   is   mainly   targeted   towards  C,   C++   or   Fortran
projects. 
Instead of writing Makefiles or autoconf/automake scripts on Unix or
Visual  Studio  projects on  Windows,  one  just  needs to  maintain  platform
independent configuration files from  which CMake will generate the previously
mentioned projects for the native build environments.\\

%
\noindent
{\sf CTest:} In order to keep  the quality of a software project high, it
is imperative to perform automated tests, which we specify in the same
simple CMake syntax.  
Every user should run 
those tests before a commit of changes or additional software
code. This ensures that all the old features are still working at
least on the user's platform.\\ 

\noindent
{\sf CDash:} This is  an open source, web-based software testing server,
which  aggregates,  analyzes and  displays  the  results  of software  testing
processes  submitted from  different clients  \cite{cdash}. We
use it to  gather information about nightly builds on various
platforms to track errors, e.g. which are platform-dependent (e.g. 32 bit vs. 64
bit). \\ 

\noindent
{\sf Trac:} This tool is an enhanced wiki and a tracking system
  for software development projects. Therewith, we can discuss
  different issues, fix upcoming development steps by assigning
  tickets, browse the {\sf Subversion} repository and look
  at the changes been made between different versions, \cite{track}.

\section{Capabilities and Features}
As mentioned, our main focus is on an efficient simulation tool for
the development of sensors, actuators and
sensor-actuator-systems \cite{makalten10:12,Kaltenbacher2011e}. Therewith, we have concentrated on FE
formulations for electromagnetics, mechanics, acoustics, thermal and
fluid flow. Furthermore, we have currently realized the
following couplings: {\it electromagnetics - mechanics - acoustics}
(electrodynamic
  loudspeaker, electromagnetic valve, electric power transformer, see, e.g.,
\cite{Kaltenbacher2011b}),
{\it piezoelectricity} (ultrasound array, surface  
  acoustic wave devices, stack actuators, see,
  e.g. \cite{hegewald08:2}), {\it electrostatics - mechanics - acoustics}
(capacitive micro-machined
  ultrasound transducers, mirror systems and microphones, see,
  e.g. \cite{kaltenbacher05:4}),  {\it fluid flow - mechanics -
    acoustics} (flow-induced sound, human phonation, see, e.g.,
  \cite{Link2009}). Most of the
  implemented algorithms are described in \cite{Kaltenbacher2007}.

Concerning special features, we may mention higher order finite
elements ($p$-FEM) and spectral finite elements 
(s-FEM) \cite{hauck06:1,Hueppe2011c,Huppe2011b,Hauck2011}, non-matching
grid techniques (Mortar FEM)
\cite{Triebenbacher2010b}, micro-macro-modeling by
homogenization \cite{kaltenbacher09:8}, advanced material models
including hysteresis \cite{kaltenbacher10:1},
inverse schemes for the precise determination of material parameters
\cite{bkaltenbacher06:3,lahmer08:1}, and topology
optimization \cite{kaltenbacher09:10,wein10:1}.

\section{CFS++ Coding Style}

There is no very strict coding style for CFS++. The main points which have been agreed upon are
\begin{itemize}
  \item No tabs only spaces.
  \item The indentation width is two characters.
  \item Source encoding is UTF-8.
\end{itemize}
