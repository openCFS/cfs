\section{Version Control System Subversion}
\label{sec:svn}

Subversion is a  free/open source version control system,  which manages files
and  directories, and  the changes  made to  them, over  time. This  allows to
recover older versions and to have different developing branches \cite{svn}.

\subsection{Working Copy Messed up by Newer Version of SVN Executable}

The svn executable  silently updates the internal format  of the working copy,b
when performing  certain operations  (e.g. add, etc.).  Therefore one  may run
into the problem, that a shared working copy is accessed by different versions
of SVN  and the older  svn executables complain,  that they cannot  access the
working copy any more.  There are two possible solutions for this problem.

First,  one  can  re-adjust  the  working  copy  version  by  using  the  {\tt
change-svn-wc-format.py}                                                 script
(cf. \url{http://subversion.tigris.org/faq.html#working-copy-format-change}).

The  second  possibility is  to  completely  abandon  the official  Subversion
executables and to use  the latest SVNKit from \url{http://svnkit.com/}, which
is a  pure Java  package and  which supports the  latest working  copy format.
Just  put its  jsvn executable  on the  PATH before  starting the  CMake build
process of CFS++.

\subsection{Relocating a Working Copy}

During  the  switch of  our  SVN  repository from  lse10  to  lse17 it  became
necessary to switch a number of local working copies to the new location. Here
is how to conveniently perform this operation:

\begin{itemize}
\item go to the working copy directory
\item issue: {\tt OLD\_REPO=\$(cat .svn/entries | grep 'CFS++/trunk')}
\item issue: {\tt svn switch --relocate \$OLD\_REPO https://lse17.e-technik.uni-erlangen.de:2001/svn/CFS++/trunk}
\item repeat analogue steps for CFSDEPS and testsuite
\end{itemize}

\subsection{Using Git in Combination with SVN}
\label{sec:git}

Git\footnote{Git home page: \url{http://git-scm.com}} is a distributed version
control  system.    It  is  an  alternative  to   the  centralized  Subversion
system. The main difference is, that there may also exist a central repository
for Git but the local working  copies of this central repository are clones of
it and are therefore repositories themselves. On these local repositories, all
operations  for which  an interaction  with  the central  repository would  be
necessary using SVN (e.g. committing changes, creating a branch, etc.)  may be
performed  locally. The  local  repository  has to  be  synchronized with  the
central one at some points  in time.  This philosophy is therefore well-suited
for  working on  laptops for  example. Fortunately  there exists  also  a link
between SVN  and Git  which allows cloning  of a  central SVN repository  to a
local   Git  repository.   Table   \ref{tab:git_materials}  contains   further
references concerning Git.

\section{CMake Build System}
\label{sec:cmake}

CMake is  a family of platform  independent tools designed to  build, test and
package  software  \cite{cmake,  Schroeder2004,  Martin2007a}.  It  is  mainly
targeted towards C,  C++ or Fortran projects. Instead  of writing Makefiles or
autoconf/automake scripts  on Unix or  Visual Studio projects on  Windows, one
just  needs to maintain  platform independent  configuration files  from which
CMake will  generate the  previously mentioned projects  for the  native build
environments. A lot of introductory and advanced material can be found on the
CMake Wiki \url{http://www.cmake.org/Wiki/CMake}.

Manually compiling a program does make sense if you only have one source file or for confirming that you supplied the right parameters to your compiler. For programs which consist of many different source files this task becomes cumbersome and error-prone. Therefore build systems have been developed which relieve the programmer of this job. The duty of the programmer just consists of specifying in a so-called make file what source files a program depends upon and how (compiler, options... ) the source files should be compiled. The standard build tool on Linux systems is GNU make. Writing make files for large software systems is again a very complex task. Getting all dependencies right can become a bold venture. Another disadvantage is the fact that make files are quite static. This means that parameters which might change for different build types must be queried from the environment.

A remedy for this problem are auto configuration systems like GNU autoconf/automake or CMake (http://www.cmake.org). Knowing the source file dependencies, such systems query the environment for platform dependent features, external libraries, etc. and generate a custom make file tailored for a certain build type. The obvious advantage is that these systems know how to write make files correctly and how to get dependencies right. A second point which makes this systems interesting is the fact that once a description of the build process has been specified for them, they can be used to generate make projects for different platforms.

In the new trunk version of CFS++ CMake has been implemented as build system. This brings a number of advantages:

\begin{itemize}
    \item OLAS/CFS++ have been merged. That means there is no more need to get the dependencies on externel libraries right. They will be set once and for all!
    \item The \#defines (e.g. USE\_LAPACK, XMLSCHEMA) have been moved from the command line of the compiler to special headers. This means that exactly those parts of the code are rebuilt that a certain \#define really affects.
    \item Due to the fact that CMake takes care about the dependencies, parallel builds are back on the road!
    \item There is no more use for the imprecise MACHINE environment variable. The build architecture will be determined during the configuration process of the build tree.
    \item Source tree and build tree are now different things. When doing an out-of-source build the source tree will not be cluttered with thousands of object files any more. 
\end{itemize}

This structure has been harmonized with the directory structure of NACS. The main directory contains the CMakeLists.txt file and it's accompanying cmake\_modules subdirectory. The CMakeLists.txt contains the main project description and handles searching for external libs and the traversal of the many subdirectories during the build. The files in cmake\_modules are meant to aid the main CMakeLists.txt in doing its job. The scripts in this directory are used to determine the distro, the compiler and to write the configuration variables to individual headers from the include directory. The CMake scripts in ctest\_scripts are meant for nightly testing using CTest/CDash. The actual C/C++/Fortran sources are located in the source directory. The share directory contains auxilliary scripts, Matlab scripts and the users' and developers' documentation and concepts.

\subsection{The build process}

The build process consists of three main steps:

\begin{itemize}
    \item Setting environment variables for compilers or paths (e.g. CC, CXX, FC, CFS\_DEPS\_ROOT,...) and creation of build directory
    \item Configuring the current build directory (e.g. ccmake SOURCE\_DIR)
    \item Building the source using make (e.g. make -j4) 
\end{itemize}

In the following subsections the steps will be described further

\subsubsection{Setting up the build directory and environment variables}

To build CFS++ we first create a build directory

\begin{verbatim}
$HOME/MY_CFS_BUILD
\end{verbatim}

Before configuring the current build directory we need to specify which compilers we want to use and where certain required directories are. The compilers (C, C++ and Fortran 95) will be set by specifying

\begin{verbatim}
export CC=gcc; export CXX=g++; export FC=gfortran
\end{verbatim}

for example. It  is of utmost importance that the correct  versions of gcc and
g++ are  available under  these names  in the system  PATH. This  is necessary
since other compilers (e.g. Intel) depend  on the GNU runtime libraries and by
default determine  the path  to these  libraries by checking  the gcc  and g++
executables in the PATH.

Please note that once a build directory has been configured with a certain set
of compilers it not advisable or  even impossible to switch to a different set
of compilers.  This is due to  the fact, that  CMake generates a lot  of flags
tailored for  each compiler. If  you want to  use a different  compiler either
create a new build directory or delete all contents of a build directory.

For systems for  which no default paths exist it is  also mandatory to specify
the following environment variables

\begin{verbatim}
export CFS_DEPS_CACHE_DIR=PATH_WHERE_SOURCES_OF_LIBS_SHOULD_GO
\end{verbatim}

Per     default     {\tt      CFS\_DEPS\_CACHE\_DIR}     points     to     {\tt
CFS\_BINARY\_DIR/cfsdeps/cache} but it is  advisable to set a global location,
since then, sources  have to be downloaded only once.

\subsubsection{Configuring the build tree}

Next we change into the build tree.

\begin{verbatim}
cd $HOME/MY_CFS_BUILD
\end{verbatim}

There we call

\begin{verbatim}
/home/shareAll/linux_bin/cmake/bin/ccmake $HOME/MY_CFS
\end{verbatim}

Then a GUI pops up and we press 'c' two times to configure.

Please note: it might be a good idea to add the CMake executable to the path.

Please note: instead of using environment variables, it is also possible to specify locations of compilers and other things to CMake on the command line.

\begin{verbatim}
ccmake -DCMAKE_C_COMPILER:PATH=/usr/bin/gcc \
       -DCMAKE_CXX_COMPILER:PATH=/usr/bin/g++ \
       -DCMAKE_Fortran_COMPILER:PATH=/usr/bin/gfortran \
       -DCFS_DEPS_CACHE_DIR:PATH=$HOME/CFSDEPSCACHE $HOME/MYCFS
\end{verbatim}

where we can type

Once the build tree has been configured a list of make targets can be aquired using

\begin{verbatim}

\end{verbatim}

\subsubsection{Specifying a different compiler}

A compiler can only be specified to CMake the first time a build tree is configured. This is due to the fact, that CMake recognizes certain compilers and internally sets some flags specific to that compilers. If the tree has been previously configured it is best to do a

\begin{verbatim}
make clean; rm CMakeCache.txt
\end{verbatim}

This should leave the build tree clean enough to configure with a different compiler. This can be done afterwards by either using

\begin{verbatim}
export CXX="mycompiler"; ccmake $HOME/MY_CFS
\end{verbatim}

or

\begin{verbatim}
export CMAKE_CXX_COMPILER="mycompiler"; ccmake $HOME/MY_CFS
\end{verbatim}

For GNU compilers the variables to set are:

\begin{verbatim}
export CC=gcc
export CXX=g++
export FC=gfortran
\end{verbatim}

For intel compilers the variables to set are:

\begin{verbatim}
export CC=icc
export CXX=icpc
export FC=ifort
\end{verbatim}

For OpenSUSE >= 11.x (cf. Sec. \ref{sec:compiler_compat_matrix})

\begin{verbatim}
source /home/shareAll/linux_bin/intel/Compiler/11.0/081/bin/iccvars.sh intel64
source /home/shareAll/linux_bin/intel/Compiler/11.0/081/bin/ifortvars.sh intel64
\end{verbatim}

or

\begin{verbatim}
source /home/shareAll/linux_bin/intel/Compiler/11.0/081/bin/iccvars.sh ia32
source /home/shareAll/linux_bin/intel/Compiler/11.0/081/bin/ifortvars.sh ia32
\end{verbatim}

For OpenSUSE < 11.x

\begin{verbatim}
source /home/shareAll/linux_bin/intel/cc(e for 64Bit!)/COMPILER_VERSION/bin/iccvars.sh
source /home/shareAll/linux_bin/intel/fc(e for 64Bit!)/COMPILER_VERSION/bin/ifortvars.sh
\end{verbatim}

\section{CFSDEPS}
\label{sec:cfsdeps}

 When  developing a  general purpose  finite element  code a  lot  of external
libraries (e.g.   BLAS, LAPACK, ARPACK, Metis\ldots)  have to be  pulled in if
the wheel is not to  be reinvented.  Many  of these libraries are  present on
modern-day Linux/Unix  distributions. If more  than the standard  GNU compiler
suite should be supported it  is however almost certain that incompatibilities
with prebuilt system libraries will arise.

 On  Windows, not  all  of the  libraries  may be  readily available.  Another
problem which arises if the Visual C++ compilers are used, is that there exist
a number  of different  C and C++  runtime environments (static  vs.  dynamic,
single-threaded  vs. multi-threaded,  debug vs.  release, etc...).   A library
compiled and linked  against one runtime environment, must  not be linked with
libraries built against a different one.  Moreover, 64-bit libraries are still
rare to find.

Due to all of the aforementioned  problems, a library may not be available and
it  has to  be  built from  source.  Therefore  we  decided to  build all  our
dependencies  in an  automated script  based  manner.  The  scripts fetch  the
sources from the web, unpack  them, perform checksum checks, patch, configure,
build and install the  libraries (cf.  Fig. \ref{fig:cfsdeps_flowdiag}).  This
way we obtain maximum platform-independence while having a well-documented and
SVN-versioned description of how to  build external packages. This approach is
not un-common among  other big software projects. To name  a few examples, the
VTK  \cite{VTKWebSite}  ships  with  and  builds libjpeg,  libtiff,  zlib  and
others. OpenFOAM\footnote{\url{http://www.openfoam.com/}} even builds it's own
GCC  and VisIt\footnote{\url{https://wci.llnl.gov/codes/visit/}} automatically
builds  a  number of  external  I/O  libs. 

Up until  spring 2012 the  build of the  external packages was  Perl-based and
took  place  at configure  time.   The  necessity  for automatically  building
external  packages has also  been recognized  by the  CMake developers  in the
meantime       and      is       supported      since       version      2.8.0
(cf.  \cite{BuildExtPackages}).  Since   the  move  of  Prof.   Kaltenbacher's
workgroup to  Vienna in  April 2012,  we now use  the CMake  external projects
mechanism, which builds all dependencies  at build time. Due to this decision,
CFS++ now requires CMake 2.8.4 to be present on the system.

Emerge\footnote{\url{http://www.gentoo.org/doc/en/handbook/handbook-x86.xml?part=2&chap=1}}
of  Gentoo\footnote{\url{http://www.gentoo.org/}}  and  the TCL  based  system
MacPorts\footnote{\url{http://www.macports.org/}} for external packages on Mac
OS X.  We used  Perl since it  runs on  all supported platforms  (Linux/Mac OS
X/Windows) and has been part of the testsuite scripts from the beginning.

Since CMake has  now builtin support for building  external projects and since
many of  the libraries, that CFS++  depends upon, now have  native CMake build
mechanisms, the  decision was  made to abandon  Perl. This should  improve the
maintainability of the code-base, especially  in the view of the upcoming NACS
2.0, since an  additional programming language can be  dropped.  Moreover, our
CMake project  had to call  the Perl build  scripts, which in turn  called the
external  CMake  builds.  This  means,  that there  were  a  lot of  potential
opportunities to introduce bugs.

Building mostly  everything from  source has the  advantage that we  can apply
patches for different compilers (GNU/Intel), different distributions or Unices
(openSUSE/Ubuntu/Mac OS  X) and different architectures  (x86, x86\_64, ia64),
without having to  rely on system libraries which  have exhibited some serious
bugs in the  past. This way we can  build upon a well tested  set of libraries
where  we  can  fix bugs  as  we  find  them.   Supporting a  different  Linux
distribution consists merely of just fixing a few minor bugs which might occur
in  the build  process. Another  advantage is  the ability  to  apply compiler
optimizations  not   only  to   CFS++  itself  but   also  on   the  dependent
libraries. And last but not least:  since the build mechanism is borrowed from
NACS an easy transition  of the features from CFS++ to NACS  and the other way
around  should  be  possible.  Even  building  most  of  the libs  on  Windows
(32-bit/64-bit) should work out of the box.

One of  the main ideas behind  CFSDEPS is that while  implementing the scripts
for  building a  package from  source the  developer implicitly  documents the
necessary  steps for  building a  library for  a given  platform. This  way it
becomes  transparent how a  lib was  built.  This  is not  given, if  a single
developer builds the library on his own machine and puts it on a network share
for others to use.  He certainly cannot  remember what he did after a year and
almost certainly errors will occur when trying to use his prebuilt binaries on
other platforms. This was however the way things were done before CFSDEPS.

Another key idea  of CFSDEPS is to have a slim  SVN repository only consisting
of  the  build  scripts,  checksum  files  and patches  needed  to  build  the
packages. This way  it is possible to build CFSDEPS/CFS++  by just getting the
SVN  working copies  of both  projects  and not  having to  copy around  large
collections  of prebuilt  libraries.  A  build machine  has  therefore minimal
requirements for installed packages  (CMake, Perl, C/C++ and Fortran compilers
and some  standard build  tools). The  idea of a  slim repository  is somewhat
counteracted by the inclusion of some  binary libs for which no source code is
available (ANSYS  and CFX customization) and  by the inclusion  of some closed
source archives (ILUPACK and SCPIP).  This compromise has been made however to
ensure a high  level of portability without having to  get these archives from
some installation directories or their authors.

Inspite of  all the  blessings of  compiling from source  one should  not keep
quiet about  some disadvantages. We  have to stick  to a certain version  of a
library to be  able to patch its  bugs. This means that we  cannot allways use
the latest version.  This is okay as  long as none of the  latest features are
required.  Another disadvantage  is  that we  have  to build  CFSDEPS even  if
developer packages of the libraries are already available on a system.

Not  all libraries  can  be built  from source  since  some of  them are  only
available in binary  form (e.g. ANSYS headers and  IO libs). Nonetheless there
exist build  scripts for  most of  them which pack  the libraries  for certain
platforms into archives.

There is  also ParaView which has been  extended by the feature  to read CFS++
HDF5  files and  OpenFOAM output  files.  It  gets also  built from  source by
CFSDEPS. Building of  Qt4 from source is also required and  takes an awful lot
of time.

\subsubsection{How CFSDEPS work}

The CFSDEPS mechanism  is tightly integrated into the  CMake configure process
of CFS++  (cf.  {\tt cmake\_modules/FindCFSDEPS.cmake} and  the files included
from there). The  download URLs for the different packages  and the files' MD5
sums are  also specified in  the aforementioned file before  the corresponding
{\tt  cfsdeps/PACKAGE/External\_PACKAGE.cmake} scripts  are included.  For more
information    on    the    CMake    external    project    build    mechanism
cf. \cite{BuildExtPackages} or the source code in {\tt ExternalProject.cmake}.


Once    the     control    has    been    handed    over     to    the    {\tt
cfsdeps/PACKAGE/External\_PACKAGE.cmake}  script   it  will  first   set  some
convenient path  variables and then specify additional  external build targets
using  the {\tt  ExternalProject\_Add()} macro.   The macro  will just  add an
additional target and  configure it during the configuration  phase, i.e. when
you type in  'cmake .'  or 'ccmake  .'. It is important to  understand that no
downloading of  sources, configuration or building of  external libraries will
take place  during the configuration step  of CFS++ anymorexs.  Just targets are
added   and  CMake  scripts   for  patching,   building  and   installing  the
corresponding packages get configured.  These  scripts however are only run at
build time, i.e. when you type in 'make'.


During the build phase,  if a package is built for the  first time, the source
archive    of    the    package    will    be    searched    for    in    {\tt
CFS\_DEPS\_CACHE\_DIR/sources/PACKAGE}.  If the archive cannot be found there,
it  will (in  most cases)  be downloaded  from an  appropriate server  and its
integrity  will be  checked  using an  MD5  checksum. If  the integrity  check
passed, the source  code will be unpacked into the  directory specified by the
variable {\tt  PACKAGE\_source}.  The  next step in  the process usually  is to
call an  extra custom\_patch step which  depends on the output  of a configured
patch script.   This way it  is possible to  take into account changes  to the
patch scripts even after the first  CFS++ configuration run. This would not be
possible    by   just    specifiying   a    {\tt   PATCH\_COMMAND}    to   {\tt
ExternalProject\_Add}, since  it would only  be run during the  first configure
run (cf. \cite{BuildExtPackages}  to see why this is).   The patch scripts get
configured      by      CMake     and      are      placed     under      {\tt
CFS\_BINARY\_DIR/cfsdeps/PACKAGE/PACKAGE-patch.cmake}.  They  are run during the
build  phase of  CFS++ and  apply the  patches depending  on OS,  platform and
compiler. For some packages also special configure, build and install commands
get  specified to the  {\tt ExternalProject\_Add}  macro.  This is usually the
case  for configure-  or make  file-based  projects.  The  built binaries  and
headers  will be installed  directly into  {\tt CFS\_BINARY\_DIR}  without being
archived into  the cache dir as  was the case  previously. This is due  to the
fact, that most  external packages inherit the build  flags from (e.g. release
ord debug) from the main CFS++ project and it would not make sense to intermix
different binaries.

CMake        will         touch        the        files         in        {\tt
CFS\_BINARY\_DIR/cfsdeps/PACKAGE/src/PACKAGE-stamp/PACKAGE-*} which correspond
to  the various  build steps,  after the  successful conclusion  of  the steps
(cf.  Fig. \ref{fig:cfsdeps_flowdiag}).   Since there  is now  way  of knowing
internal dependencies of the external  libs, the library will not get rebuilt,
even if its  source code gets modified by hand. Therefore,  it is necessary to
delete the stamp  file corresponding to a certain build step,  if a bug should
be  fixed inside an  external package.  The safest  way to  make sure,  that a
certain   package   gets   rebuilt   is   to   delete   its   directory   {\tt
CFS\_BINARY\_DIR/cfsdeps/PACKAGE}  and all  the files  it install  and running
{\tt cmake . \&\& make} afterwards.

\begin{itemize}
\item To get started you should have at least CMake 2.8.4 and a working set of

\subsubsection{pardiso}  PARallel DIrect  SOlver\cite{Schenk2004, Schenk2006a,
Schenk2006b, Schenk2007}. Needed by ipopt and (ilupack).

\noindent Project Homepage: \url{http://pardiso-project.org}

To put it frankly, Pardiso, being a closed source project, is a software engineering nightmare! That is due to three main facts:

\begin{description}
  \item[Licensing]
    \subitem The Pardiso of Schenk introduces a new license policy with every second version or so. For example, version 4.0 had a license duration of one year. Version 4.1 has a duration of just one month instead! This is the reason for the next point.
    \subitem License files are not portable across different versions. A license file valid for version 4.0 is not valid for version 4.1 anymore, since it would break the 4.1 license policy. That means that every Pardiso user has to re-request a license file every month. This is especially tedious, if automated nightly test are to be performed.
    \subitem Since version 4.1.2 Schenk changed his license policy in such a way, that a free evaluation license which is even only valid for 30 days cannot be refreshed. The web server says something like this: {\emph An evaluation license key for user name strieben and host name sedici has been generated before.
You cannot download more than once with the same license data.} An academic license now costs 500 CHF, which we are certainly not willing to pay. Supporting Schenk's Pardiso has therewith become an enterprise, which I (Simon) am not prepared to support any more. I will take the nightly test for Pardiso offline from now on.
  \item[Non-backward compatible API]
    \subitem The API of the pardisoinit\_ and pardiso\_ functions changed from version 3.x (as in MKL) to version 4.x, but no backward compatible replacements have been provided.
    \subitem There is no decent way of obtaining the API version from a binary library of Pardiso. There exists no API function for querying the API version and no C or Fortran headers are provided, which would contain that information.
    \subitem The only way of obtaining the version information is to link a version 3 or 4 source to a given library and to check according to buffer overflows or segfaults which API is correct (cf. cmake\_modules/CheckPardisoAPIVersion.cmake). This way of testing is remarkably dirty, unstable and not guaranteed to work with future versions.
    \subitem Due to the fact, that we also support MKL, the check is obligatory, since MKL still has API version 3. If only one version of the API would be supported in our source (cf. sources/OLAS/external/pardiso/pardisosolver.cc), linking against a different library could provoke undefined bahaviour or even segfaults.
  \item[Distribution policy]
    \subitem Old versions of libraries get removed from the download servers, whenever a new version is released. This means, that one has no chance of obtaining a library for a given license file, if a new release has been made. This problem is still somehow acceptable for a closed source library like Pardiso, but it is still very annoying since it breaks automated tests.
    \subitem The set of supported platforms changes from release to release. It is absolutely not understandable what the policy for supported platforms is. The support for Mac OS X has been dropped from 4.0 to the initial release of 4.1 for example. Also the compiler versions (GCC, Intel) change from release to release and only one GCC version is supported per release. On the other hand the GCC version for 32-bit and 64-bit Linux is different for example. In conclusion it would appear that they build their libraries by hand and do not even have a certain platform policy.
\end{description}

Due to the aforementioned reasons the build\_pardiso.pl script needs to be modified with every new release. Fortunately the libraries all share the same base URL, at least for a given release. Therefore just the BASE\_URL and the SLIB (serial lib) and PLIB (parallel lib) variables need to be changed for the different platforms for a new release.

In order for the configuration tests to work, a proper license file (pardiso.lic) has to be obtained and the environment variable PARDISO\_LIC\_PATH has to set to the path containing it.

\subsubsection{MKL}

The versions  9.x and 10.x are supported  by CFS++ at the  moment.  The search
code is  defined in cmake\_modules/FindIntelMKL.cmake. For  all common systems
(lseXX, altix, woody,  etc.)  all search paths are  predefined there.  MKL 9.x
will only  be used  if version 10.x  is not  available, or if  the environment
variable  MKL\_ROOT\_DIR  has  been  set  to its  path  prior  to  configuring
CFS++. The setting of this  environment variable also precedes the settings of
MKL\_ROOT\_DIR\_DEFAULT         which          gets         defined         in
cmake\_modules/platform\_defaults\_*.cmake.

The  path   given  in  MKL\_ROOT\_DIR   should  point  to  a   directory  like
/opt/intel/mkl/10.0.5.025   in  the   case   of  a   standalone   MKL  or   to
/opt/intel/Compiler/11.0/081/mkl in the case where  MKL is a part of the Intel
11.x compiler suite.

The  OpenMP  version of  MKL  will  be used  no  matter  what  the setting  of
USE\_OPENMP in  CMake is. This means that  CFS++ would try to  use the maximum
number of processors  available on a machine to do  linear algebra. To prevent
this,  the CFS++  start script  bin/cfs (share/scripts/cfs)  will set  the the
environment variables OMP\_NUM\_THREADS and MKL\_NUM\_THREADS to 1 by default,
if  none of  them have  been set  to a  different value  previously.   If only
MKL\_NUM\_THREADS has not  been set previously it will be set  to the value of
OMP\_NUM\_THREADS. To check this do the following before running CFS++

export CFS\_SCRIPT\_DEBUG=1

The cfs script will issue a warning if OMP\_NUM\_THREADS and MKL\_NUM\_THREADS
are set to a different value.

ATTENTION: Please keep in mind that CFS++  will try to use all processors of a
machine, if the bin/ARCH\_STRING/cfsbin binary is used directly!

\subsection{Strategies for Fixing Build Problems on new Compiler Versions}

By  default the  compiler switches  for CFS++  are set  to a  very restrictive
setting concerning  warnings and  errors. Warnings in  the code  are therefore
treated as errors, as they usually indicate portability problems. We have done
this to force some more discipline out of the developers and to make sure that
the code stays in a compilable and maintainable state.

The drawback  of this decision is, that  the same strict rules  are applied to
headers from  the external libraries in  CFSDEPS. This can cause  trouble if a
new  compiler version interprets  the C++  standard in  a stricter  manner and
complains about  problems, which  have not been  problems before.   Because of
this,  it becomes  sometimes necessary  to fix  such problems  in the  code of
external libs.  We prefer to apply our  fixes to the external  libs instead of
switching off special warnings in the  build flags for CFS++, since the latter
approach would weaken our strict rules for clean code.

The most notable problems which can occur include the following:
\begin{itemize}
\item Warnings occuring in external headers (e.g. Boost headers).
\item Linker problems due to  builtin/intrinsic functions of GCC which are not
available  in   Intel  compilers  (cf.    \cite{WikiIntrinsic,  GCCExtensions,
MSVCIntrinsics}  and  Intrinsics   Reference  and  the  Building  Applications
$\rightarrow$    gcc   Compatibility   $\rightarrow$    gcc   Interoperability
$\rightarrow$ gcc Built-in Functions section in \cite{IntelCOnlineDoc}).
\item Build problems in our own code.
\item  Problems occuring due  to changed  console output  of the  command line
utilities used by the CFSDEPS  build scripts. (fix: change regular expressions
in CFSDEPS/utils/perl/ checkforcmdlinetools.pl)
\item Problems due  to missing command line tools  (fix: install packages from
distro).
\item Problems due  to changed Subversion working copy  formats (fix: put jsvn
from latest SVNKit \url{http://svnkit.com/} on PATH).
\end{itemize}

\subsubsection{Fix problem in build directory and create patches for CFSDEPS}
\url{https://svn.boost.org/trac/boost/wiki/Guidelines/WarningsGuidelines}

\subsection{Stepping through the CFS++ testsuite with CTest}

It is now possible to step through the test cases in the CFS++ testsuite using
the CTest system.  CTest is an  extension to CMake which provides a convenient
framework for defining software  tests (cf. \cite{Martin2010}). In conjunction
with CDash  (cf. Sec. \ref{sec:cdash})  automated dashboards can  be generated
from  nightly builds of  a software  for example,  or even  unit tests  can be
generated.

\subsection{Adding test cases to the testsuit}

To   add   a  test   case   just   specify   another  SUBDIRS()   command   in
\$\{TESTSUIT\_DIR\}/CMakeLists.txt and copy the CMakeLists.txt from one of the
other subdirectories to the new subdirectory. Make sure that the filenames are
written   in  the   same  way   like  their   containing   subdirectory  (e.g.
\$\{TESTSUIT\_DIR\}/optimization/piezosimp/output\_idbc/:  output\_idbc is the
subdirectory  and  the  files  have  to  be  named  like  output\_idbc.xml  or
output\_idbc.mesh).    There   also   have   to   be   two   reference   files
\{SUBDIR\_NAME\}.plot.ref.dat  and  \{SUBDIR\_NAME\}.h5ref. Afterwards  ccmake
has to be called in the desired directory as follows:

\begin{verbatim}
${CFS_BUILD_DIR}/cfs-test> ccmake .
\end{verbatim}

Press [c] to configure the CMakeCache.txt file. Was it configured successfully
one can press  [g] to update the  makefile for ctest. Now the  TESTSUIT can be
executed:

\begin{verbatim}
ctest [-R SELECTED_PART]
\end{verbatim}


\section{CDash}
\label{sec:cdash}

This is an  open source, web-based software testing  server, which aggregates,
analyzes and displays the results of software testing processes submitted from
different clients \cite{cdash}.  We use it to gather information about nightly
builds  on various  platforms. This  way we  can track  down errors  which are
platform-dependent (e.g. 32 bit vs.  64 bit) more efficiently.

The  central dashboard  for  CFS++ is  located  on the  development server  at
\url{https://lse17.e-technik.uni-erlangen.de:2001/cdash}.
